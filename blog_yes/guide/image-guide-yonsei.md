# 연세예스내과의원 블로그 이미지 가이드 (신규 프레임 방식)

> 이 가이드는 **연세예스내과의원**용 신규 배경/프레임 이미지를 쓸 때의 제작 기준이다.
> 기존 `image-guide.md`(클립보드 오렌지 프레임 방식)는 그대로 두고, 이 문서는 신규 배경/프레임 에셋 체계를 다룬다.
> **2026-07-15 대개편**: 실제 게시된 연세예스내과 블로그 9장(간경변증/건강검진/영양수액 각 3장)을 레퍼런스로
> 받아, 좌우분할 표/체크리스트 방식을 폐기하고 **리스트형(Template A) / 본문형(Template B)** 2종 템플릿 +
> **우상단 코너 라벨**(시리즈 문구 + 원형 순번) 체계로 전면 재구성했다. §2·§3·§4가 이번 개편의 핵심이다.

## 2. 본문 프레임 — 실측 좌표 및 2종 템플릿 (2026-07-15 대개편)

### 2-0. 프레임 자체를 수정했다 — 헤더 라벨/로고 위치 이동

기존에는 `fraim1/2/3.png`에 이미 박혀 있는 "YONSEI YES INTERNAL MEDICINE" 헤더 텍스트와 하단 로고 위치를
그대로 두고 콘텐츠 영역만 그 사이에 끼워 넣었다. 2026-07-15에 **레퍼런스 이미지와 비교해 이 두 요소를
카드 구석 쪽으로 직접 이동**시켰다 (PIL로 해당 영역을 크롭 → 흰색으로 원위치 지우기 → 새 위치에 paste,
`fraim1/2/3.png` 3장 모두 동일하게 적용, 원본은 스크래치패드에 백업):

| 요소 | 기존 위치 (bbox) | 새 위치 |
|------|------|------|
| 헤더 라벨 "YONSEI YES INTERNAL MEDICINE" | x 321–1075, y 505–530 | 좌상단으로 이동, 새 bbox 좌상단 **(159, 380)** |
| 하단 로고 "연세예스내과" | x 958–1796, y 2352–2533 | **60px 아래로** 이동 (새 top ≈ 2412) |

이 이동으로 카드 상/하단에 남는 여백을 콘텐츠 영역 쪽으로 되돌려줄 수 있게 됐다 (`CONTENT_TOP`/`CONTENT_BOTTOM` 참고).

### 2-1. 카드/캔버스 실측 좌표

`fraim1/2/3.png` 3장 모두 흰 카드 지오메트리는 동일(탭 색상만 다름), 2756×2756 기준, 픽셀 스캔으로 직접 실측:

```
CARD_LEFT, CARD_TOP, CARD_RIGHT, CARD_BOTTOM = 140, 335, 2615, 2626   # 흰 카드 자체의 경계
TEXT_LEFT   = 321   # 기존 헤더 라벨 x (참고용 — 현재는 제목 정렬 기준으로 쓰지 않음, §2-3 참고)
CONTENT_TOP    = 450    # 이동된 헤더 라벨(y~380-405) 아래 여백을 반영해 위로 당김
CONTENT_BOTTOM = 2370   # 이동된 로고(y~2412~) 위 여백을 반영해 아래로 늘림
RIGHT_MARGIN = TEXT_LEFT - CARD_LEFT   # 181 — 카드 우측에도 대칭 적용
TEXT_RIGHT = CARD_RIGHT - RIGHT_MARGIN  # 2434
FULL_W = TEXT_RIGHT - TEXT_LEFT         # 2113 — 제목이 정중앙 정렬되는 전체 폭
```

- 콘텐츠 영역 높이 = `CONTENT_BOTTOM - CONTENT_TOP` = 1920px (2026-07-15 개편 전 1720px보다 200px 늘어남)
- 이 영역 안에서 제목 + (Template A/B) 콘텐츠를 배치한다

### 2-2. 제목 — 항상 정중앙 정렬 + 고정 크기 (좌측 정렬 폐기)

레퍼런스 조사 초반에는 제목을 좌측 정렬(헤더 라벨과 같은 x)로 시도했으나, **최종적으로는 "제목은 어느
한쪽에 치우치지 말고 정중앙 정렬, 좌/우 배치는 본문·리스트에만 적용"으로 확정**했다 (2026-07-15).

- 제목은 `FULL_W`(2113px) 전체 폭 기준 **가로 중앙 정렬**, `<div class="title">`을 별도로 렌더링해
  `CONTENT_TOP`에 붙인다 (Template A/B 공통)
- 제목 실제 필요 높이(`title_h`)는 Playwright로 먼저 측정(`measure_height`)한 뒤, 본문/리스트는
  `body_top = CONTENT_TOP + title_h + TITLE_GAP`(TITLE_GAP=70px) 아래부터 시작한다 — 즉 제목 줄 수(1줄/2줄)에
  따라 본문 시작 y가 자동으로 밀린다.
- 제목 아래 리스트/일러스트만 좌/우로 배치되고, 제목 자체는 그 두 컬럼 폭 전체를 가로질러 중앙에 위치한다.

### 2-3. 우상단 코너 라벨 (신규, 레퍼런스에서 확인)

레퍼런스 9장 전부에 카드 위쪽 다크 네이비 배경 스트립(카드 시작 전 y<335 영역)에 **시리즈 문구 + 원형
순번**이 우측 정렬로 들어가 있었다. 탭(주황/파랑/핑크)은 로고 색상 순환과 무관하게 항상 좌상단에 있고,
라벨은 그 오른쪽 다크 영역에 겹치지 않게 배치된다.

- 문구: 포스팅의 원 소스 문구(예: 사용자가 준 "[주안동수액병원] 영양 수액 맞기 전에 알면 좋은 정보!"에서
  병원명을 뺀 "영양 수액 맞기 전에 알면 좋은 정보!")를 그대로 쓰고, 뒤에 **원형 순번**(①②③... 대신 PIL로
  직접 그린 얇은 원 + 아라비아 숫자)을 붙인다. 순번은 프레임 색상 순환(fraim1→2→3→1→2...)과 무관하게
  **포스팅 내 이미지 순서(1,2,3,4,5...)** 를 그대로 따른다.
- PIL로 프레임 위에 직접 그린다(Playwright 렌더링 대상이 아님, 썸네일 제목 그리는 방식과 동일 계열):
  ```
  label_color = (199, 203, 207)         # 옅은 회색 — 다크 배경 위 텍스트/원
  font = Pretendard-SemiBold, 44px      # 문구 텍스트
  circle_font = Pretendard-SemiBold, 40px  # 원 안 숫자
  circle_d = 66px, stroke width = 4px
  cy = 220                              # 다크 스트립(0~335) 안에서 수직 중심
  right_margin = 40px                   # 카드 우측 경계(CARD_RIGHT)로부터 원의 우측 여백
  ```
  문구는 원의 왼쪽에 22px 간격을 두고 우측 정렬로 이어 붙인다.

### 2-4. Template A — 리스트형 (번호 리스트 + 우측 하단 일러스트)

레퍼런스의 "1. 2. 3. ..." 번호 리스트 스타일. 항목이 5~6개인 체크리스트/원인·주의사항 나열에 적합.

```
LIST_LEFT  = 260    # 카드 왼쪽 여백(140)의 약 2배 지점 — 제목과 무관하게 리스트만 별도 좌측 정렬 기준
illust_w   = 1000   # 일러스트 폭 — 카드 중앙 부근까지 도달하도록 (2026-07-15 확정, 기존 760px에서 확대)
gap_col    = 113    # 리스트-일러스트 사이 간격
illust_h   = min(1480, CONTENT_BOTTOM - body_top - 20)   # 제목과 겹치지 않도록 상한을 body_top 기준으로 제한
text_w     = illust_left - gap_col - LIST_LEFT
```

- 리스트/일러스트 좌우 배치는 **포스팅 내 Template A 이미지 등장 순서로 자동 교대**한다 (2026-07-16 추가) —
  1번째 Template A 이미지는 리스트 왼쪽/일러스트 오른쪽, 2번째는 반대로 일러스트 왼쪽/리스트 오른쪽, 이하 반복.
  같은 배치가 계속돼서 지루해 보이는 것을 막기 위함이며, 코너 라벨(§1)은 좌우와 무관하게 항상 카드 우상단 고정이다.
- 리스트는 항상 **좌측 정렬된 텍스트**로 렌더링하고(번호는 프레임 탭 색상과 동일한 accent 색으로 굵게 표시,
  텍스트와 18px 간격), 배치될 컬럼(왼쪽 or 오른쪽)의 x좌표만 위 교대 규칙에 따라 바뀐다.
- 리스트/일러스트 모두 **`body_top`(제목 하단)~`CONTENT_BOTTOM`(로고 상단) 구간의 세로 중앙**에 맞춰 배치한다
  (2026-07-16 변경 — 과거엔 리스트가 위쪽, 일러스트가 아래쪽에 붙어 서로 어긋나 보였다). 리스트는 실제 렌더링된
  텍스트 높이(`measure_height`로 측정)를, 일러스트는 `illust_h`를 각각 같은 `center_y = (body_top + CONTENT_BOTTOM) / 2`에
  맞춰 중앙 정렬하고, 구간을 벗어나지 않도록 상/하단으로 clamp한다.
- **줄바꿈은 반드시 단어 단위로** — `.list li { word-break: keep-all; overflow-wrap: normal; }`을 지정하지
  않으면 한 글자 때문에 줄바꿈될 때 단어 중간이 잘려 다음 줄로 넘어간다 (2026-07-15 발견·수정). 번호(`.num`)는
  `flex-shrink:0`, 텍스트 span은 `min-width:0`을 줘야 flex 컨테이너 안에서 정상적으로 줄바꿈된다.
- 항목 개수가 5개 이하면 리스트 항목 간 gap 46px, 6개 이상이면 30px로 좁힌다.

### 2-5. Template B — 본문형 (제목 → 인물 일러스트 → 문단)

레퍼런스의 "설명형" 스타일. 정의·개념 설명, 서술형 문단에 적합.

```
avail_h = CONTENT_BOTTOM - body_top
```

1. 먼저 **일러스트 높이 0인 상태**로 "제목 아래 이미지자리+문단" 전체를 측정해 `non_image_h`(문단 자연 높이 등)를 구한다
2. `img_h = clamp(avail_h - non_image_h - 10, 280, 950)`, `img_w = img_h * 0.95` — **문단 텍스트가 항상 먼저
   확보되고, 남는 공간만큼만 일러스트 높이가 정해진다** (문단이 길면 일러스트가 작아지고, 짧으면 커짐)
3. 확정된 `img_h`/`img_w`로 최종 렌더링 — 제목(중앙) → 일러스트(중앙) → 문단(중앙, `word-break:keep-all`)
   순서로 세로 스택, 전부 가로 중앙 정렬

⚠️ **제목 2배 확대(§3) 이후 겪은 문제와 해결 과정 (2026-07-15)**:
- 제목을 190~220px로 키우면서 Template B의 이미지 높이를 고정값(예 0.56×폭)으로 두었더니, 본문 폰트를
  50% 확대(60~83px)한 조합에서 **문단 마지막 줄이 카드 하단을 벗어나 잘리는 문제**가 발생했다 (여백/제목/이미지가
  고정폭을 다 먹어버려 문단에 남는 세로 공간이 부족했음).
- 해결: 위 1~3단계처럼 **문단 높이를 먼저 측정하고 이미지 높이를 나머지로 역산**하는 방식으로 바꿔, 문단이
  항상 넘치지 않게 보장했다. 대신 일러스트가 너무 작아지는 문제가 생겨(최소 296px까지 줄어듦), 사용자 요청으로
  본문 폰트를 60px로 재조정(83px→60px)하고 `CONTENT_TOP`/`CONTENT_BOTTOM`을 늘려(§2-1) 여유 공간을 추가 확보했다.

### 2-6. 공통 — scale 관련 원칙은 계속 유효

과거 버전(좌우분할 표/체크리스트 방식)에서 확정된 아래 원칙은 이번 개편 이후에도 그대로 적용된다:

- **폰트 크기는 항상 §3에 지정된 값이 실제 렌더링 크기의 상한이다.** 콘텐츠가 넘칠 때만 축소하고, 절대
  확대(1.0 초과)하지 않는다 — 그래야 포스팅 내 모든 이미지의 글씨 크기가 시각적으로 동일하게 보인다.
- 정렬 기준(`align-items`)을 바꿀 때는 `transform-origin`(또는 이번처럼 스택 구조 자체)도 같은 기준점으로
  맞춰야 콘텐츠가 잘리지 않는다.
- "영역을 최대한 채운다"는 목표는 폰트 확대가 아니라 **레이아웃(칼럼 폭, 리스트 좌측 위치, 일러스트 크기)**
  으로 달성한다.

## 3. 타이포그래피

| 요소 | 크기 | 폰트 |
|------|------|------|
| 메인 타이틀 | **190px (확정, 2026-07-15 재조정)** | Pretendard ExtraBold, **font-weight: bold 명시**, 항상 정중앙 정렬 |
| 본문 텍스트(Template B 문단) | **60px (확정)** | Pretendard SemiBold |
| 리스트(Template A) 항목 | **60px (확정, 본문과 동일)** | Pretendard SemiBold, 번호만 accent 색 + ExtraBold |

- **이 값은 여러 차례 조정을 거친 최종값이다** (2026-07-15, 레퍼런스 기반 개편 과정에서 재확정):
  1. 처음엔 좌측 정렬 제목(110px)·본문 55px 체계였으나(구버전, 아래 히스토리 참고), 레퍼런스 비교 후
     "제목은 치우치지 않게 정중앙, 크기는 현재의 2배(→220px)"로 변경 요청을 받아 반영
  2. 본문도 "50% 키워달라"는 요청으로 55px→83px로 확대했으나, **Template B에서 일러스트 공간이
     296px까지 줄어드는 문제**가 발생 (§2-5 참고) → "본문 사이즈를 현재의 80%로"라는 요청으로 83px→66px로 재조정
  3. 이후 라벨/로고 위치 이동(§2-0)으로 생긴 여백을 반영해 제목·본문을 각각 **190px / 60px**로 최종 하향 조정
  (제목 220→190, 본문 66→60) — 여백이 늘어난 만큼 콘텐츠 영역(§2-1)이 넓어져 폰트를 무리하게 키우지
  않고도 일러스트 공간을 충분히 확보할 수 있게 됐다.
- **제목이 좁은 폭에서 2줄로 넘어가는 경우**, 자동 줄바꿈에 맡기지 말고 `<br>` 태그로 줄바꿈 위치를
  의미 단위로 직접 지정한다 (예: `이런 분들은<br>특히 주의하세요` — "주의하세"/"요"처럼 단어 중간에서 끊기지 않도록).
- **Template A 리스트 항목의 줄바꿈도 단어 단위로 고정한다** (`word-break: keep-all`) — §2-4 참고.
- **제목(타이틀)은 무조건 굵게 처리한다.** Pretendard ExtraBold 폰트 지정만으로 끝내지 않고 CSS에 `font-weight:bold`를 명시적으로 추가한다 — 폰트 fallback(예: Malgun Gothic) 상황에서도 항상 굵게 나오도록 하는 안전장치
- 강조 요소(표 헤더, 스텝 번호, 인용구, 카드 굵은 글씨 등)는 Pretendard ExtraBold + `font-weight:bold` 사용
- **Pretendard 폰트는 이 PC에 사용자 폰트로 설치됨** (관리자 권한 불필요)
  - 설치 위치: `%LOCALAPPDATA%\Microsoft\Windows\Fonts\Pretendard-*.ttf`
  - 레지스트리 등록: `HKCU\Software\Microsoft\Windows NT\CurrentVersion\Fonts`
  - 사용 가능한 굵기 패밀리명: `Pretendard`(Regular/Bold), `Pretendard Thin/ExtraLight/Light/Medium/SemiBold/ExtraBold/Black`
  - CSS에서는 `font-family: 'Pretendard ExtraBold', ...` 처럼 굵기별 패밀리명을 직접 지정해야 함 (variable font가 아니라 static 굵기별 파일이라 font-weight 숫자로는 매칭 안 됨)
  - 다른 PC에서 이 파이프라인을 돌릴 경우 동일하게 폰트 설치 필요 (`orioncactus/pretendard` GitHub 릴리즈 `Pretendard-1.3.9.zip` → `public/static/alternative/*.ttf`)

## 4. 이미지(일러스트) 자리 및 프롬프트

- 분할에서 텍스트가 차지하지 않는 나머지 영역이 **일러스트 삽입 자리**
- 이 파이프라인은 이미지를 직접 생성하지 않고, **영어 프롬프트만 산출**한다 → 사용자가 별도로 생성해 자리에 합성
- 프롬프트 공통 규칙:
  - 스타일: **Flat Vector Illustration**
  - 문구에 반드시 `Freepik "flat illustration" / Storyset-style stock illustration` 포함
  - `soft pastel color palette`, `no text` 포함 권장
  - 장면은 해당 콘텐츠 주제와 연결되는 단순한 상황 묘사로 작성

### 4-1. 배경/인물/사이즈 규칙 (2026-07-15 레퍼런스 기반 재확정)

> ⚠️ **"행위가 있을 때만 인물"이라는 구버전 규칙은 폐기한다.** 실제 게시된 레퍼런스 9장을 확인한 결과,
> 성분 나열·효과 설명처럼 행위가 없는 개념 설명 자리에도 인물이 거의 매번 등장했다 (예: "영양수액이란?" 자리에
> IV 폴대 옆에 서 있는 의료진, "효과가 있을까요?" 자리에 저울을 든 고민하는 인물). **실제 운영 중인 자산과
> 스타일을 맞추는 것이 우선**이므로, 아래처럼 인물 포함을 기본값으로 바꿨다.

1. **배경은 가급적 넣지 않는다 — 넣더라도 흰색 단색으로 통일한다** — 색이 들어간 배경(파스텔 톤 벽지 등)을 쓰지 않고, `isolated on a plain white background`, `no background scenery/elements`로 명시해 배경 디테일(창문·가구·풍경·색상 배경 등)을 배제한다.
2. **원칙적으로 인물을 포함한다** (2026-07-15 재수정) — 개념 설명이든 리스트든 관계없이 사람이 등장하는 장면으로 구성하고, 그 개념/감정을 인물의 자세·표정·소품으로 표현한다 (예: 저울+체크마크/물음표를 든 고민하는 인물 = "효과가 있는지 판단하기 어렵다"는 개념, 팔에 멍이 든 인물 = 부작용). 사물만으로 표현하는 것은 실제로 인물로 표현하기 매우 부자연스러운 경우(예: 단순 아이콘화된 수치 비교표)에 한한 예외로 취급한다.
3. **인물은 전신 또는 상반신으로 표현한다** — 얼굴/손 클로즈업 같은 부분 크롭은 지양한다.
4. **이미지 자리의 가로/세로 비율을 프롬프트에 명시한다** — Template A(리스트형)의 일러스트 자리는 세로로 긴 직사각형이므로 `portrait orientation, vertical composition`, Template B(본문형)는 정사각형에 가까운 비율이므로 이에 맞는 `aspect ratio`를 명시하고, 실제 계산된 `image size {w}x{h}px`를 프롬프트 끝에 함께 적는다.
5. **여백은 상하좌우 5pt만 남기고 인물/오브젝트가 프레임을 꽉 채우도록 한다** (2026-07-16 추가) — 인물을 캔버스
   중앙에 작게 그리고 넓은 흰 여백을 남기는 구도를 금지한다. 프롬프트에 반드시 다음 문구를 포함한다:
   `subject fills the frame edge-to-edge with only a ~5px margin on all four sides, no extra
   padding or empty space around the subject, tightly cropped full-bleed composition`.
   (5pt 자체는 AI 이미지 생성기가 정밀하게 지킬 수 있는 단위가 아니므로, "거의 여백 없이 꽉 채운다"는 의도를
   전달하는 영어 문구로 치환해 지정한다 — 실측 여백이 아니라 프롬프트 지시 문구임에 유의.)

### 4-2. 프롬프트 예시는 주제별 파일 참고

과거 버전(좌우분할·객체 위주)의 예시 프롬프트는 이번 대개편으로 모두 무효화되어 삭제했다. 실제 사용 예시는
항상 최신 작업의 `output/[yymmdd_주제]/illustration_prompts.txt`를 참고한다 — 이 파일에 매 주제마다
Template A/B 구분, 한글 장면 설명, 인물 중심 영어 프롬프트, 정확한 `image size`가 함께 기록된다 (§6 참고).

## 5. 썸네일 (`thumb nail.png`, 1024×1024)

- 디자인: 스프링노트 스타일, 점선 가로줄 9개(y=140~815, 간격 ~84px), 우측에 컬러 탭 3개, 하단에 로고
- **제목 텍스트는 하단 중앙에 배치** — 상단 2/3는 일러스트 이미지를 위해 비워둔다
- 좌표(검증된 값):
  ```
  CARD_CENTER_X = 525   # 2026-07-16 재재실측 — 점선 줄무늬 자체의 좌우 끝(x≈170~879, 9줄 픽셀 스캔) 중앙 기준.
                         # 카드 흰 배경 전체 폭 기준(44~925 → 485)은 점선이 실제 차지하는 폭보다 넓어 여전히 치우쳐
                         # 보인다는 피드백으로 점선 폭 기준으로 재조정 (과거 545/485 모두 폐기)
  LINE_Y_START  = 650
  LINE_SPACING  = 115
  MAX_WIDTH     = 660   # 좌측 여백 확보를 위해 카드 폭(850)보다 좁게 제한
  TEXT_COLOR    = #1A2340
  ```

- 2줄 제목 기준으로 테스트했고, 줄 수가 늘어나면 `LINE_Y_START`를 위로 당겨야 함 (로고 영역 y>900 침범 금지)

### 5-1. 제목 텍스트 규칙 (2026-07-15 확정)

- **썸네일 제목은 포스팅 전체 제목(H1)을 그대로 쓰지 않는다.** 예: 포스팅 제목이 "온열질환 현황과 국가지침"이어도
  썸네일에는 "온열질환"처럼 핵심 단어만 축약해서 쓴다.
- 축약형은 자동으로 확정하지 않는다 — **매번 사용자에게 물어보거나, 사용자가 직접 지정한 문구를 그대로 사용한다.**
  (예: "썸네일 제목은 뭐로 할까요? 자동 축약하면 'OO'인데 이대로 괜찮을까요?" 형태로 확인)
- 확인받은 문구만 §5의 `TITLE_LINES`(1~2줄)로 렌더링한다.

### 5-2. 썸네일 일러스트 이미지 (2026-07-15 확정, 2026-07-15 좌표 재검증)

- 상단 2/3 빈 공간에 **일러스트 이미지 1장**을 정중앙 배치한다 — **제목 바로 위, 카드 정중앙**이 기준 위치.
- 이 일러스트도 본문 이미지와 동일한 §4-1 규칙을 따른다: **배경 없음(넣어야 한다면 흰색 단색 배경으로 통일)**,
  §4-1 2번 기준으로 인물/사물 여부 판단, **인물을 쓸 경우 전신 또는 상반신**, 이미지 자리 방향/사이즈를 프롬프트 끝에 명시.
- 자리 좌표 (**픽셀 스캔으로 실측 완료** — `thumb nail.png` 카드 백지 영역 기준):
  ```
  THUMB_IMAGE_BOX = (left=116, top=110, right=886, bottom=600)   # width 770 × height 490
  ```
  - 카드 백지 영역의 실제 가로 중심은 **x≈501** (y=110/150/200/500/580에서 픽셀 스캔으로 확인) — 제목의
    `CARD_CENTER_X`=545는 왼쪽 스프링 링을 피하려고 의도적으로 오른쪽으로 옮긴 값이라 이미지 중심에는 그대로 쓰면 안 된다.

  - `LINE_Y_START`(650)보다 위, 카드 상단 백지 시작(y≈73)보다 아래 영역
- 프롬프트 작성 시 본문 프롬프트와 마찬가지로 `output/[yymmdd_주제]/illustration_prompts.txt`에 함께 저장한다

## 6. 작업 순서 — 프롬프트 저장 → 사용자 이미지 삽입 → 합성 → 최종 조립 (2026-07-15 확정)

일러스트가 필요한 자리가 있는 주제는 **최종 파일(`final.md`/`final.html`)을 만들기 전에** 사용자가 이미지를
직접 생성해 끼워 넣는 단계를 반드시 거친다. 순서는 다음과 같다.

0. **썸네일 제목 확인 (필수, §5-1)**: 썸네일에 쓸 축약 제목을 사용자에게 물어보거나 사용자가 지정한 문구를
   확인받는다. 포스팅 전체 제목을 임의로 줄여서 확정하지 않는다.
1. **프레임 렌더링**: 프레임+텍스트만 합성한 `body-{n}.png`를 만든다 (일러스트 자리는 비워둠). 썸네일도
   확인된 축약 제목으로 텍스트만 렌더링한다 (일러스트 자리는 비워둠).
2. **사용자 검수 (필수, 2026-07-15 추가)**: 프롬프트를 작성하기 전에 1단계에서 만든 `body-{n}.png`/`thumbnail.png`
   (프레임+텍스트만, 그림 자리는 비어있는 상태)를 사용자에게 먼저 보여주고 승인을 받는다.
   - 소제목 문구, 표/체크리스트 내용, 좌우·상하 분할 방향, 프레임 색상 순서 등에 문제가 없는지 확인
   - 사용자가 수정을 요청하면 다시 1단계로 돌아가 재렌더링 → 다시 검수
   - 승인 전에는 3단계(프롬프트 파일 저장)로 넘어가지 않는다
3. **프롬프트 파일 저장 (필수)**: 본문 일러스트 프롬프트와 썸네일 일러스트 프롬프트를 채팅 응답이나
   scratchpad에만 남기지 말고, **반드시 `output/[yymmdd_주제]/illustration_prompts.txt` 파일로 먼저 저장한다.**
   - `[body-N] 소제목 — Template A/B, 방향/사이즈` 헤더 + 프롬프트 전문 형식 유지 (§4-2 참고)
   - 썸네일 프롬프트는 `[thumbnail] 축약 제목 — 가로/세로 자리, 770x490px` 헤더로 같은 파일에 포함 (§5-2 참고)
   - **각 프롬프트 위에 어떤 장면인지 한글로 1~2문장 요약을 반드시 함께 적는다** (영어 프롬프트만으로는
     사용자가 어떤 그림이 나올지 바로 파악하기 어렵기 때문). 채팅 응답에도 동일한 한글 요약을 먼저 안내한다.
   - 이 파일이 저장된 시점에서 사용자에게 알리고, 이미지 생성·전달을 기다린다.
4. **사용자 이미지 확보**: 사용자가 각 프롬프트로 이미지를 직접 생성해 파일 경로를 전달한다.
5. **합성**: 전달받은 이미지를 각 `body-{n}.png`의 빈 이미지 자리에 맞춰 크롭/리사이즈 후 합성하여
   `output/[yymmdd_주제]/images/body-{n}.png`로 저장한다 (§2의 `image_box` 좌표 계산 로직 재사용).
6. 합성이 모두 끝난 뒤에만 CLAUDE.md Step 4(assembler)로 넘어가 `final.md`/`final.html`을 만든다.

즉 **"프레임 렌더링 → 프롬프트를 output 폴더에 파일로 저장 → 사용자 이미지 확보 → 합성 → 최종 조립"**
순서를 반드시 지킨다. 프롬프트를 파일로 먼저 남겨야 사용자가 최종 조립 전 어느 시점에든 이미지를
검토·교체할 수 있다.
